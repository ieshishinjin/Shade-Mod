# Minecraft Galgame 剧情系统 — 通用融合设计文档

## 核心目标

构建一个通用的 Galgame 剧情框架，能与现有任何游戏功能（Camp、采集、建造、战斗、贸易等）深度联动，且未来新功能可无缝接入。AI 生成引擎能够感知所有已对接系统的状态，生成跨系统的复合剧情。

---

## 一、设计哲学

### 1.1 核心思想

剧情框架是"心脏"，游戏系统是"手脚"。心脏不关心手脚具体长什么样，只关心手脚能做什么。

- 剧情框架本身不依赖任何具体游戏系统
- 任何游戏功能都可以通过"适配"被剧情系统调用
- Quest 目标不限定类型，可以是"占领营地"、"采集物品"、"击杀Boss"、"建造建筑"等任意目标
- AI 能读取所有已对接系统的状态，生成复合型剧情

### 1.2 三层架构

| 层级 | 职责 | 依赖方向 |
|------|------|----------|
| 表现层 | 对话框 UI、Quest 追踪 HUD、CG 展示、日记/图鉴、画廊 | 依赖中层 |
| 逻辑层 | 剧情引擎、Quest 系统、分支控制、条件判断、触发器 | 依赖底层接口，不依赖具体系统 |
| 系统层 | Camp、采集、建造、战斗、日记、图鉴等具体游戏功能 | 向上提供适配接口 |

---

## 二、通用 Quest 系统

### 2.1 设计思路

Quest 系统不关心"具体要做什么"，只关心：
- 这个目标是什么类型（占领？收集？击杀？）
- 完成条件是什么
- 当前进度是多少

**核心思路**：Quest = 若干个 Objective 的组合，每个 Objective 都是可插拔的。

### 2.2 Quest 数据结构

| 字段 | 说明 |
|------|------|
| quest_id | 唯一标识 |
| quest_name | 显示名称 |
| quest_description | 任务描述 |
| quest_objectives | 目标列表，每个目标包含类型、目标ID、数量、显示文本 |
| quest_rewards | 奖励内容，包含经验、物品、剧情标志 |
| on_quest_complete | 完成后跳转的节点 ID |
| on_quest_fail | 失败后跳转的节点 ID |

### 2.3 Objective 类型扩展表（开放式）

| 分类 | Objective 类型 | 说明 | 来源系统 |
|------|---------------|------|----------|
| 营地 | OCCUPY_CAMP | 占领指定营地 | Camp系统 |
| 营地 | ATTACK_CAMP | 对营地造成防御伤害 | Camp系统 |
| 营地 | DEFEND_CAMP | 保护营地不被攻破 | Camp系统 |
| 营地 | UPGRADE_CAMP | 提升营地等级 | Camp系统 |
| 资源 | COLLECT_ITEM | 获取指定数量物品 | 原版/模组 |
| 资源 | CRAFT_ITEM | 合成指定数量物品 | 原版合成 |
| 战斗 | KILL_MOB | 击杀指定数量实体 | 原版战斗 |
| 战斗 | KILL_BOSS | 击败特定Boss | Boss系统 |
| 探索 | REACH_LOCATION | 到达指定坐标区域 | 原版探索 |
| 社交 | TRADE_VILLAGER | 完成指定次数交易 | 原版交易 |
| 建造 | BUILD_STRUCTURE | 建造指定建筑 | 建造系统 |
| 经验 | REACH_LEVEL | 达到指定经验等级 | 原版经验 |
| 自定义 | CUSTOM | 任意扩展目标 | 任何系统 |

## 三、游戏系统适配层

### 3.1 适配层的作用

适配层是剧情框架与具体游戏系统之间的桥梁。任何一个游戏系统想被剧情调用，只需要实现一套标准接口：

- 告诉我你是谁 → 系统唯一标识
- 你能做什么 → 支持的 Objective 类型列表
- 执行这个操作 → 剧情调用你的某个功能
- 你现在什么状态 → 供 AI 读取和剧情条件判断

### 3.2 复合 Quest 示例

一个 Quest 可以同时涉及多个系统：

**Quest：收复边境**
- Objective 1（Camp系统）：占领西边哨站
- Objective 2（采集系统）：收集50个原木用于修缮
- Objective 3（战斗系统）：击杀10只掠夺者

## 四、剧情脚本格式

### 4.1 节点类型

| 节点类型 | 用途 |
|----------|------|
| DIALOG | 显示角色对话，点击继续 |
| CHOICE | 显示多个选项供玩家选择，不同选择走向不同分支 |
| QUEST_START | 发布 Quest（可包含跨系统的复合 Objective） |
| QUEST_UPDATE | 更新 Quest 进度显示 |
| QUEST_COMPLETE | 完成 Quest，发放奖励 |
| CONDITION | 检查某个条件（如营地状态、物品数量），自动跳转 |
| EVENT | 触发游戏内事件（传送、给予物品、切换BGM等） |
| END | 关闭对话框，结束当前章节 |
| AI_GENERATE（预留） | 调用 AI 动态生成后续内容 |

### 4.2 节点字段

| 字段 | 说明 |
|------|------|
| id | 节点唯一标识 |
| type | 节点类型 |
| speaker | 说话人名称 |
| portrait | 立绘资源路径 |
| text | 对话文本内容 |
| options | 选项列表（仅 CHOICE 类型） |
| quest | Quest 数据（仅 QUEST_START 类型） |
| condition | 条件判断（仅 CONDITION 类型） |
| event | 事件数据（仅 EVENT 类型） |
| next | 下一个节点 ID |
| on_quest_complete | Quest 完成后的跳转节点 |
| on_quest_fail | Quest 失败后的跳转节点 |

### 4.3 条件系统

| 条件类型 | 说明 | 示例 |
|----------|------|------|
| CAMP_STATUS | 某营地是否被占领/摧毁 | 西边哨站已被占领 → 触发胜利剧情 |
| ITEM_COUNT | 玩家是否有足够物品 | 背包有10个钻石 → 触发特殊对话 |
| KILL_COUNT | 是否击杀足够怪物 | 击杀100只僵尸 → NPC称赞玩家 |
| PLAYER_LEVEL | 玩家是否达到指定等级 | 等级≥20 → 解锁高级 Quest |
| FLAG_CHECK | 是否完成过某事件 | 已完成序章 → 解锁第一章 |
| LOCATION | 玩家是否在指定区域 | 到达村庄中心 → 触发剧情 |

## 五、AI 剧情生成器

### 5.1 AI 的感知能力

AI 生成剧情时，能够"看到"当前游戏世界的完整状态：
- 所有已对接系统的状态摘要（Camp、资源、进度等）
- 玩家的位置、等级、生命值
- 当前正在进行的 Quest
- 已完成的历史剧情
- 已解锁的日记和图鉴条目

### 5.2 AI 的约束

- Quest Objective 类型必须从已对接系统支持的类型列表中选择
- Camp ID、物品 ID 等必须使用游戏中真实存在的 ID
- 对话内容需符合 Minecraft 世界观
- 生成内容需为安全可控的 JSON 格式

### 5.3 自动生成

系统支持在以下条件自动触发 AI 生成剧情：
1. 首次进入游戏 → 生成开场剧情
2. 完成剧情/任务 → 生成后续剧情
3. 世界等级提升 → 生成新挑战
4. 玩家行为累积 → 生成相关剧情
5. 定时触发 → 日常/随机事件

自动生成由 AutoStoryGenerator 管理，每次生成后至少有 30 分钟冷却。

### 5.4 AI 生成流程

1. 系统收集当前游戏状态（所有适配器提供摘要）
2. 系统收集剧情上下文（历史、当前节点、玩家输入）
3. 系统构建 Prompt，发送给 AI
4. AI 返回 JSON 格式的剧情内容
5. 系统解析并校验（检查引用的 ID 是否存在、类型是否合法）
6. 将生成内容转换为剧情节点，插入当前剧情流
7. 展示给玩家

## 六、模块划分总览

| 模块 | 职责 | 是否依赖具体游戏系统 |
|------|------|---------------------|
| 剧情引擎 | 加载脚本、推进节点、管理状态 | ❌ |
| 对话系统 | 显示对话框、打字机效果、选项交互 | ❌ |
| Quest 系统 | 管理 Quest、追踪进度、判断完成、发放奖励 | ❌ |
| 条件系统 | 检查各种条件决定分支 | ⚠️ 依赖适配器 |
| 适配器注册表 | 管理所有已注册的游戏系统适配器 | ❌ |
| AI 生成器 | 收集状态、构建 Prompt、解析 AI 返回 | ⚠️ 依赖适配器 |
| 日记系统 | 记录剧情关键信息、NPC、地点 | ❌ |
| 图鉴系统 | 记录生物、物品、方块发现 | ❌ |
| 具体系统适配器 | 对接 Camp、采集、战斗等具体系统 | ✅ |

## 七、开发路线

### Phase 1：基础框架
- 设计剧情脚本格式规范（节点类型、字段定义）
- 实现剧情引擎核心（加载、解析、节点跳转）
- 实现玩家进度存储
- 实现剧情脚本热加载

### Phase 2：UI 系统
- 实现对话框（立绘、说话人、文字、打字机效果）
- 实现选项按钮（点击后分支跳转）
- 实现 Quest 追踪 HUD
- 实现对话框的打开/关闭/跳过动画

### Phase 3：Quest 系统
- 实现 Quest 数据结构（包含多个 Objective）
- 实现 Quest 进度追踪（每个 Objective 独立进度）
- 实现 Quest 完成判断
- 实现 Quest Reward 发放
- 实现 Quest 日志

### Phase 4：条件与事件
- 实现条件系统（支持多种条件类型）
- 绑定游戏事件到 Quest 进度更新
- 实现剧情触发事件

### Phase 5：命令与交互
- 实现 /story 命令系列
- 实现 NPC 右键触发剧情
- 实现区域进入触发剧情
- 实现物品获得触发剧情

### Phase 6：适配器层
- 设计适配器标准接口（`SystemAdapter`）
- 实现 6 种适配器（Camp、Inventory、Combat、Villager、System、Passive）
- 实现适配器注册表

### Phase 7：AI 生成器
- 设计 AI 生成器接口（`AiProvider`）
- 实现 DeepSeek / Ollama / Claude / OpenAI 兼容 Provider
- 构建完整的游戏状态摘要
- 实现 AI 返回内容的解析和校验

### Phase 8：多分支结局
- 根据玩家 Flag 条件解锁不同结局
- 画廊条目支持 condition 字段

### Phase 9：CG 展示系统
- 全屏 CG 插画展示，支持淡入淡出

### Phase 10：世界事件系统
- 随机世界事件触发（流星雨、怪物攻城、贸易车队）
- 每位玩家独立冷却

### Phase 11：NPC AI 对话
- 潜行+右键触发村民或命名生物的 AI 即兴对话

### Phase 12：安全防护
- AI 生成的剧情节点安全白名单

### Phase 13：日记/图鉴系统（v0.0.7）
- 实现日记和图鉴数据模型（JournalEntry / BestiaryEntry）
- 实现日记管理器和图鉴管理器（单例 + JSON 持久化）
- 实现网络数据包（S2C 推送 + C2S 请求）
- 实现双标签页 GUI（按 J 打开）
- 实现自动解锁事件钩子（击杀解锁图鉴、完成剧情解锁日记）
- 剧情菜单添加日记按钮

### Phase 14：数据包加载脚本（v0.0.7）
- `StoryEngine.loadScripts()` 改用 `ResourceManager.listResources()`
- 支持从数据包 `data/<命名空间>/story/*.json` 加载剧情脚本
- 兼容现有模组资源 `assets/shade/story/*.json`
- 数据包脚本优先级高于模组内置脚本

### Phase 15：多世界支持（v0.0.7）
- 移除 `StoryEventHandler` 和 `CampEventHandler` 中的 `OVERWORLD` 维度过滤
- 剧情系统、Quest 追踪、世界事件、触发器在所有维度中均可工作
- 据点系统：仅在主世界自动生成新据点，已有据点在任意维度可管理

### Phase 16：触发器管理 GUI（v0.0.7）
- 实现触发器管理图形界面（TriggerManageScreen）
- 左栏列表 + 右栏详情，支持删除操作
- 新增三个网络数据包（列表请求/列表响应/删除请求）
- 剧情菜单底部添加「⚙ 触发器管理」按钮
- 仅 OP 玩家可删除触发器
